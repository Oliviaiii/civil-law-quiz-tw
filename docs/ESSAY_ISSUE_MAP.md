# 申論題考點地圖與跨考試關聯規格

狀態：第一批資料層已完成 120/120 題草稿覆蓋；申論考點總覽（章節層級）已上線，葉節點爭點詳情、繁中名稱 catalog、搜尋、跨題關聯與真人法律複核尚未完成
記錄日期：2026-07-23
主要使用者：準備司法特考四等法院書記官的考生

## 1. 目標

把已收錄的官方申論題整理成可查找、可統計、可跨考試比較的「申論考點地圖」，讓使用者能回答：

- 歷屆申論題考過哪些主題與爭點？
- 同一爭點在哪些年度、科目及考試重複出現？
- 題目之間是相同爭點、同法條不同問題，還是原則與例外的對照？
- 某爭點近年是否持續出現？
- 點進一個爭點後，有哪些官方原題可以一起練習？

本功能只能呈現「已納入且已複核的官方試題曾考過什麼」，不能宣稱涵蓋法律上所有可能爭點，也不能把歷屆頻率包裝成未來命題預測。

## 3. 內容邊界

### 3.1 官方資料

下列內容只能來自考選部官方資料，匯入時不得改寫：

- 考試名稱、等別、類科與適用類科。
- 年度、科目、試卷、官方題號與配分。
- 題目原文及官方試題網址。
- 官方公告、應考須知與科目資訊。

申論題通常沒有考選部標準擬答，因此不得產生或顯示「官方答案」欄位的假內容。

### 3.2 本站整理

下列內容屬於本站編輯資料，畫面與資料模型都必須和官方資料清楚分開：

- 命題主旨。
- 主要爭點與次要爭點。
- 子題拆分。
- 相關法條、裁判、解釋及法規版本說明。
- 題目之間的相同、相近或對照關係。
- 整理信心、人工複核狀態與複核日期。

第三方教材、教科書、補習班或題庫網站只能協助研究爭點，不得複製或近似改寫其完整詳解。這一階段也不提供 AI 擬答；命題主旨與爭點標籤不是擬答。

## 4. 分類模型

### 4.1 統一爭點樹

爭點使用穩定 ID 建立受控分類，不以畫面顯示文字作為關聯鍵。基本層級為：

```text
科目
└─ 編／章／制度
   └─ 爭點
      └─ 必要時再分子爭點
```

例如：

```text
民法
└─ 總則
   └─ 意思表示
      ├─ 錯誤
      ├─ 詐欺
      ├─ 脅迫
      └─ 撤銷與第三人保護
```

每個爭點節點至少應能表達：

```ts
type EssayIssue = {
  id: string;
  subject: string;
  parentId: string | null;
  label: string;
  aliases: string[];
  description: string;
  statuteRefs: string[];
  status: "draft" | "reviewed";
};
```

`aliases` 只用來協助搜尋與整理舊稱，不可為同一概念重複建立多個 ID。爭點名稱過粗或過細時，先在民法試作批次調整，不要在所有科目各自發明分類尺度。

### 4.2 每題整理資料

官方原題維持在既有題庫；新增的編輯資料以 `questionId` 關聯，不直接覆寫官方題幹。建議資料至少包含：

```ts
type EssayAnnotation = {
  questionId: string;
  gist: string;
  primaryIssueIds: string[];
  secondaryIssueIds: string[];
  subparts: {
    label: string;
    issueIds: string[];
  }[];
  lawVersionNote: string | null;
  confidence: "low" | "medium" | "high";
  reviewStatus: "draft" | "reviewed";
  reviewedAt: string | null;
  references: string[];
};
```

規則：

- `gist` 用一至兩句說明題目主要在考什麼，不寫成完整擬答。
- 每題至少一個主要爭點；沒有充分依據時不得用大量模糊標籤充數。
- 複合題以 `subparts` 保留題目內部結構，再分主要與次要爭點。
- 涉及舊法、修法或裁判見解變更時，必須填寫 `lawVersionNote`。
- `references` 優先保存官方法規、裁判、解釋或可供人工複核的研究來源。
- 未人工複核的資料只能標為 `draft`，前台不得偽裝成已確認內容。

正式實作前可依既有 `Question`、`Reference` 型別調整欄位名稱，但不得破壞上述資訊邊界。

### 4.3 題目關聯

能由共同爭點推導的關聯不重複手寫；只有無法由標籤充分表達的關係才建立顯式資料：

- `same-core-issue`：同一核心爭點。
- `same-statute-different-question`：同法條但問題不同。
- `contrast`：原則與例外、肯定與否定情形的對照題。
- `prerequisite`：先備概念或前後階段關係。
- `substantive-procedural`：實體法與程序法交叉。
- `law-change`：修法或見解變更前後比較。

純文字相似度或 AI 分群只能用來提出候選關聯，不能未經複核直接發布。

## 5. 跨考試資料範圍

擴充其他考試前，先建立官方試卷 manifest，不直接把搜尋到的題目貼進現有題庫。候選優先順序為：

1. 司法官及律師第二試。
2. 與法院書記官科目高度重疊的司法特考其他等別或類科。
3. 高普考法制、法律廉政等相關類科。
4. 其他經確認對法院書記官備考有直接價值的考試。

是否實際納入某一考試，依下列條件評估：

- 與法院書記官現行科目及命題深度的相關性。
- 考選部官方原題是否可穩定取得並逐年核對。
- 試卷、類科及題號能否正確識別。
- 整理與人工複核成本。
- 是否會讓主要使用者被低相關題目淹沒。

同一官方試卷若適用多個類科，只保存一份題目，另以適用類科欄位表示；不得因不同類科或不同匯入批次重複計算。

## 6. 統計口徑

考點地圖至少同時提供「題目數」與「出現年度」，避免只顯示一個容易誤讀的熱門度分數。

- 同一題對同一爭點最多計一次，即使該題多個子題都涉及該爭點。
- 共用試卷只計一次，不因適用多個類科重複加總。
- 一題可分別計入不同爭點，但畫面應能回到原題核對。
- 主要爭點與次要爭點應可分開統計。
- 顯示最近五年、全部收錄年度及各考試分布時，必須同時揭露資料涵蓋範圍。
- 「近期常見」只能描述已觀察到的歷屆分布，不得改寫為必考、命題機率或押題結論。

## 7. 預期畫面

### 7.1 申論考點總覽

- 依科目、章節、爭點、考試、年度與複核狀態篩選。
- 顯示各爭點的官方題目數、出現年度及涉及考試。
- 支援分類樹與關鍵字搜尋。
- 清楚顯示目前收錄範圍及最後資料更新日期。

### 7.2 爭點詳情

- 爭點說明、別名、所屬分類與相關法規。
- 所有相關官方題目，依主要爭點、次要爭點及關聯類型分組。
- 年度與考試分布。
- 相近、對照、修法前後及實體／程序交叉題。

### 7.3 申論題卡

- 保留官方題目與來源連結。
- 顯示「本站整理」的命題主旨與爭點標籤。
- 可由標籤進入爭點詳情或切換到相關題目。
- 未複核、涉及舊法或資料涵蓋不完整時提供明確提示。
- 與既有草稿、計時及自我檢核功能並存。

## 8. 建議實作順序

### 階段 0：來源與結構盤點

- 重新計算現有法科申論題數與 ID，確認 120 題基線。
- 檢查每題考試、年度、科目、題號、題幹與 `sourceUrl`。
- 決定分類檔案位置、型別與產生流程。
- 建立外部考試 manifest 草案，但不匯入大量題目。

### 階段 1：民法 26 題試作

- 先建立民法爭點樹。
- 完成 26 題命題主旨、主次爭點、子題、法規版本及參考來源。
- 人工複核分類粒度、同義詞、統計結果與關聯類型。
- 只有在民法樣板穩定後，才將結構套用到其他科目。

### 階段 2：完成現有 120 題

- 依序處理刑法、行政法、民事訴訟法、刑事訴訟法。
- 每次只處理一科或可驗收的小批次。
- 每批完成後更新覆蓋率、來源紀錄與交接狀態，不累積大量未複核草稿。

### 階段 3：前台考點地圖

- 加入總覽、爭點詳情、篩選及題卡入口。
- 搜尋索引只加入必要的摘要與標籤，不把整份新增資料塞進首頁初始 bundle。
- 保持 GitHub Pages 純靜態、按需載入與 PWA 更新機制。

### 階段 4：跨考試試作

- 先選一種高相關考試及一個科目做完整年度樣本。
- 驗證官方來源、ID、共用試卷去重、題目關聯及前台篩選。
- 確認使用價值與維護成本後，再決定是否擴大其他考試。

## 9. 驗收與防退化

資料測試至少應覆蓋：

- 官方題目 ID 唯一，並能從每筆整理資料回查原題。
- 收錄範圍內每題都有官方來源網址。
- 每筆已發布整理資料都有命題主旨及至少一個主要爭點。
- 所有爭點 ID 都存在於統一分類，沒有孤兒或循環父節點。
- 同一題、同一爭點的統計不重複。
- 共用試卷不因類科重複計數。
- 涉及舊法或修法的題目有版本註記。
- `draft` 與 `reviewed` 在資料及畫面上可區分。
- 搜尋、科目篩選、年度篩選及題目深連結仍可正常運作。
- 首頁效能預算、靜態建置、PWA 資料版本與舊進度相容性不退化。

內容驗收至少應抽查：

- 命題主旨是否只摘要問題，而非偷偷生成擬答。
- 主爭點是否真的是得分核心，次要爭點是否過度標記。
- 同義爭點是否被統一，而非因用語不同拆成多個節點。
- 跨考試關聯是否有法律理由，而不只是題幹文字相似。
- 頻率及趨勢文案是否忠實呈現資料範圍。

## 10. 不在本階段處理

- 自動產生申論題擬答或評分。
- 宣稱本站爭點清單等於完整考試範圍。
- 根據歷屆頻率提供必考預測或押題機率。
- 未經人工複核就大量發布 AI 標籤。
- 為此功能引入帳號、後端資料庫、Server Action 或外部 LLM API。
- 將國文作文、公文混入法律爭點統計。

## 11. 待決策事項

進入實作前，需在對應 Issue 或 PR 明確記錄：

- 外部考試第一批要選哪一種考試、哪些年度與科目。
- 民法試作後確定的分類粒度及爭點命名慣例。
- 整理資料檔案的最終位置與是否需要產生式索引。
- `reviewed` 由誰確認，以及多人複核時如何保存紀錄。
- 前台第一版先做分類列表，或同時加入視覺化年度趨勢。

## 12. 接手工作檢查表

下一位協作者開始前應：

1. 閱讀 `README.md`、`CONTRIBUTING.md`、`docs/SUBJECT_EXPANSION.md` 與本文件。
2. 先確認目前分支、題數、題目 ID、工作區變更及官方來源，不沿用過期數字。
3. 建立或更新 GitHub Issue，寫清楚本次科目、年度、資料批次與驗收範圍。
4. 只處理一個可完整研究及複核的批次。
5. 保留官方原題與本站整理的資料邊界。
6. 執行資料、lint、靜態建置及 Playwright 驗證；資料產生器造成的變更一併檢查。
7. 未經明確指示不得自行部署。
8. 交接時記錄已完成題目、待複核題目、來源、測試與已知風險。
